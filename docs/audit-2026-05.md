## 3. Schema do Banco

> Schema completo em `prisma/schema.prisma` (1290 linhas). Resumo abaixo.

### Modelos definidos (28 no total)

#### Tenants & auth

- **`Company`** — tenant raiz; toda outra tabela tem FK pra cá. Carrega branding (logo, slug, address, phone, email, contactName) usado nas exports.
- **`Role`** — papel global (Admin, etc). Não é per-tenant.
- **`User`** — credenciais + Gmail OAuth tokens. FK pra Company + Role.
- **`Module`** — registro dos módulos do app pro sistema de permissão.
- **`UserModulePermission`** — matriz user × module × (canView/Create/Edit/Delete).

#### Lead intake

- **`Client`** — empresa cliente do tenant (companyName, address, etc).
- **`ClientContact`** — contatos por cliente (name, email, phone, isPrimary).
- **`Bid`** — bid recebido por email/manual. Status: new/qualified/sent_to_takeoff/won/lost. FK pra Client.
- **`BidLink`** — URLs externas extraídas do email (plans, portal, meetings).
- **`BidDocument`** — anexos do bid (PDFs, etc).
- **`BidStatusHistory`** — audit log de mudanças de status.
- **`BidAiAnalysis`** — análise IA de prequalificação (distance score, work type score).
- **`BidExtraction`** — auditoria de chamada Claude que extraiu o bid do email.
- **`SystemSetting`** — KV per-tenant (defaults de margem, distância, work types).

#### Takeoff

- **`Project`** — agrupa todo trabalho de takeoff de uma obra. FK opcional pra Bid. Tem campos de IA-1 (stories, durationWeeks, siteConditions, requiredEquipment, winterRisk, permitChecklist).
- **`ProjectDocument`** — PDFs/plantas/specs anexados ao Project. Tem `anthropicFileId` cached pra Phase 1.
- **`ClassificationFolder`** — árvore de pastas pra organizar templates.
- **`ClassificationTemplate`** — biblioteca per-tenant de classifications reutilizáveis.
- **`Classification`** — instância de classification num Project (qty, uom, scope). Tem campos Togal (togalId, togalFolder) + Phase 1 (sourceAnalysisRunId, aiConfidence, quantityBasis, needsTogalVerification).
- **`TakeoffImport`** — audit log de imports externos (togal/manual/csv).

#### Estimate — catálogo de referência

- **`Division`** — top-level trade (Siding, Roofing, Rough Carpentry, etc). Per-tenant.
- **`Region`** — mercado geográfico (MA, NH, RI). Drives labor rate selection.
- **`LaborTrade`** — categoria de mão-de-obra (Carpenter Rough, Tile Setter, etc).
- **`LaborRate`** — taxa horária por trade × region × shopType (open_shop/union). Low/Avg/High.
- **`ProductivityEntry`** — MH per unit pra um escopo. Match via matchCode (ELFCS, EL02) ou matchKeywords. **Funciona como service catalog.**
- **`MaterialType`** — categoria hierárquica de material (Fiber Cement Siding, Lumber).
- **`Material`** — item de catálogo (name, sku, uom, low/avg/highCents, wastePercent, supplier, supplierUrl).
- **`Assembly`** — recipe combinando labor + material num único line item (não está no caminho de auto-pricing inicial).
- **`AssemblyComponent`** — peça de uma assembly (qty per unit + refs/inline labor + material).
- **`CostFactor`** — multiplier aplicado no estimate (Boston Metro +15%, Prevailing Wage +35%, HVHZ +18%, Winter +8%). appliesTo: labor/material/overhead.

#### Estimate — operacional

- **`Estimate`** — uma por Project (`@unique projectId`). Status: draft → in_pricing → pricing_done → submitted_to_client → won/lost/cancelled. Carrega config de pricing (region, shopType, mhRangeMode), margens (markup, overhead, GC, contingency, salesTax), envelopeSf, assumptions, validForDays.
- **`EstimateLine`** — uma linha por Classification ou ad-hoc. Snapshots imutáveis (nome/qty/uom/laborTrade/materialBreakdown). Tem AI tracking (suggestedByAi, aiConfidence, needsReview, source: togal-id/togal-prefix/ai-classified/ai-derivative/manual). Hierarchy via parentLineId pra derivative lines.
- **`EstimateCostFactor`** — snapshot de cost factors aplicados no estimate (autoApplied flag).

#### IA / sugestões / Phase 1

- **`DerivativeCostRule`** — regras per-tenant que geram "implicit" cost lines (fastener allowance, etc). Trigger: matchCode ou divisionId. Formula JSON discriminada.
- **`Suggestion`** — fila genérica "AI propõe, Andre aprova". Type: productivity/material/derivative_rule/productivity_rate_change/equipment_rule. Status: pending → approved/rejected/archived. **A IA NUNCA escreve em master tables direto.**
- **`ProjectAnalysisRun`** _(Phase 1)_ — uma execução do scope-analyst. Espelha BidExtraction (audit completo). Status: pending → accepted/partially_accepted/rejected/failed.
- **`ClassificationCalibration`** _(Phase 1)_ — uma row por item proposto pela IA num run, mesmo se rejeitado. Snapshot do que IA propôs vs o que ficou. Suporta Togal verification cross-check.

### Multi-tenant — como está implementado

- **Padrão `companyId` em todas as tabelas operacionais** (linha-by-linha tenancy, não schema-separado nem RLS).
- Toda query API valida `companyId === ctx.companyId` via `requireAuth()` de `@/lib/permissions`.
- O context auth carrega `{ userId, email, role, companyId, companyName }` via NextAuth JWT session.
- `Role` é global (uma tabela compartilhada). `User` é per-tenant via FK pra Company.
- Master tables **per-tenant também**: cada Company tem suas próprias Divisions, Materials, ProductivityEntries, LaborRates, etc.
- Único Single tenant em produção: **JMO Carpentry** (slug `jmo`). Test Company existe no banco mas não é usada.

## 6. Estado de cada módulo

### 🟢 Auth / Login

NextAuth v5 beta com Credentials provider. Pages em `/login` e `/register`. JWT sessions, bcrypt para senha. Gmail OAuth também integrado (separado, pra captura de leads). `auth.ts` na raiz de `src/`. **Funcional em produção** (Andre logado como Admin no JMO Carpentry).

### 🟢 Multi-tenant (organizações/empresas)

Implementado via padrão **`companyId` em toda tabela operacional + master**. Não usa schemas separados nem RLS — confia em filtro explícito em cada query. `requireAuth()` retorna `companyId` na ctx; `canDo()` checa permissão no escopo. Atualmente single tenant em prod (JMO Carpentry); arquitetura está pronta pra mais tenants.

### 🟢 Permissões e roles

Models: `Role` (global) + `Module` + `UserModulePermission` (matrix user × module). Helper `canDo(ctx, moduleSlug, action)` em `lib/permissions.ts`. Slugs em uso: `bid`, `takeoff`, `estimate`. Admin sempre passa. **Funcional**, mas a UI de gerenciamento de permissões é simples (em `/users/[id]` provavelmente).

### 🟢 Master Data — Clientes

Models `Client` + `ClientContact`. Pages `/clients` e `/clients/[id]` com CRUD completo via `/api/clients`. Address/city/state/zip nos campos. **Funcional.**

### ⚪ Master Data — Funcionários

**Não existe.** O `User` model é apenas pra autenticação no app — não tem dados de RH (cargo, salário, hire date, payroll info, dependentes, etc). Não há model `Employee` separado.

### ⚪ Master Data — Subcontratados

**Não existe.** Sem model dedicado pra subcontractors. Implícito: `LaborTrade` agrupa categorias de mão-de-obra mas não captura entidades subcontratadas com contatos, contratos, ratings.

### ⚪ Master Data — Fornecedores

**Não existe enquanto entidade.** O model `Material` tem campos `supplier String?` e `supplierUrl String?` como texto livre — sem normalização em uma tabela `Supplier` com contatos, payment terms, lead times, ratings.

### 🟢 Master Data — Materiais

Models `Material` + `MaterialType` (hierárquico via `parentId`). Page `/admin/reference/materials` + API endpoints CRUD. Carrega supplier (texto livre), low/avg/high cents, wastePercent, lastPricedAt. **Funcional.** Vinculado a Division opcional.

### 🟢 Master Data — Serviços

Implementado via `ProductivityEntry` (MH per unit por escopo). Page `/admin/reference/productivity` + API CRUD. Tem matchCode (ELFCS, EL02) + matchKeywords pra resolver classifications automaticamente. **Funcional.** Complementado por `LaborTrade` + `LaborRate` (taxa horária por região + shopType).

### 🟢 Lead Intake (captação de leads)

Pipeline completo:

1. **Gmail OAuth** + sync periódico (`/api/gmail/sync`) — busca emails de bid.
2. **Claude extrai** dados estruturados → `BidExtraction` (audit completo: tokens, custo, prompt usado).
3. **Andre aceita** → `BidExtraction` vira `Bid` via `/api/bids/from-extraction`.
4. **Prequalification** (`BidAiAnalysis`) — distance score, work type score, recommendation.
5. **Status flow**: new → qualified → sent_to_takeoff → won/lost.
6. UI completa em `/bids` (lista + detail + new manual).

**Funcional em produção.**

### 🟢 Estimate / Bid (orçamento)

Pipeline produtivo end-to-end:

1. **Takeoff manual ou via Togal** (XLSX import) cria `Classification` rows.
2. **`accept-estimate`** dispara pricing determinístico:
   - `togal-resolver` faz match L1/L2 (matchCode → ProductivityEntry).
   - Fallback `priceClassification` faz fuzzy match.
   - Cria `EstimateLine` com snapshots imutáveis.
3. **OH&P calculados FLAT** (GC% + Overhead% + Profit% sobre direct cost; não cascateado).
4. **Cost factors** auto-aplicados por região.
5. **AI helpers**: IA-1 (Project Context), IA-2 (Hidden Cost Detector), IA-3 (Re-price all batch), per-line AI suggestions.
6. **Sync from Takeoff** — banner detecta classifications órfãs adicionadas no Takeoff após accept e oferece sincronizar.
7. UI: `/estimates/[id]` com 14 colunas editáveis (Code/Description/Trade/Qty/UOM/MH/u/$/hr/Total MH/Labor $/Mat $/u/Material $/Subtotal/AI/actions).

**Funcional em produção.**

### 🟡 Proposal (geração de proposta)

Estado parcial:

- 🟢 **Excel export funcional** — duas versões (interno completo c/ OH&P breakdown vs cliente sem OH&P, markup multiplicativo embutido). Logo, header com Company info + Client info side-by-side, signature block, validity (default 30d).
- ❌ **PDF export NÃO implementado** — botão existe na UI mas marca "soon". Lib alvo era `@react-pdf/renderer`, não foi instalada.
- ❌ **Versão "envio ao cliente" formal** (com numbering, lock, audit, e-mail send) ainda não está implementada.
- 🟡 **Phase 1 Project AI Analyst** — backend completo (schema + endpoints + run.ts + Files API + polling). UI: panel de listagem de runs e botão de trigger funcionam; **drawer formal de revisão / accept flow ainda não implementado** (página de detail é read-only). Próximo trabalho da fase.

### ⚪ Contracts

**Não existe.** Sem model `Contract` no schema. `Estimate.status` chega até `won` mas não há o passo "vira contrato" com clauses, milestones, e-signature, change orders.

### ⚪ WBS (Work Breakdown Structure)

#### Lógica de negócio dentro de componentes UI

Onde acontece:

- **`estimates/[id]/page.tsx`** (1518 LOC) — calcula `rollupTotals` no client (importa de `lib/estimate-pricing.ts`, OK), mas tem inline: pricing factor math, OH&P breakdowns, derivative parent/child grouping, sync from takeoff dedup.
- **`bids/[id]/page.tsx`** (1478 LOC) — tem inline: prequalification status transitions, Gmail attachment download, geocoding triggers, document version reconciliation.
- **`clients/[id]/page.tsx`** (831 LOC) — primary contact promotion logic + ClientContact CRUD inline.
- **`classifications-sidebar.tsx`** (818 LOC) — toggle scope, edit row, delete row, dialog state, totals — tudo num arquivo. Helper já extraído pra `lib/classifications-actions.ts` mas só `toggleClassificationScope`; o resto continua inline.

Mitigação parcial: `lib/` tem 19 módulos com lógica pura bem isolada (pricing engine, togal-resolver, derivative-rules-engine, gmail, geocoding, ai-\* helpers). Mas os pages consomem isso direto + adicionam mais lógica em cima.

#### Duplicação de lógica

- **Cost calculation duplicada**: `lib/claude-client.ts → costForUsage()` (Opus 4.7 only) vs `lib/ai/scope-analyst/cost.ts → calculateCostCents()` (Opus + Sonnet fallback). Pricing tables não compartilham source of truth.
- **Anthropic response parse** repetido em 4 pipelines: `bid-extraction.ts`, `ai-project-context.ts`, `ai-hidden-costs.ts`, `estimate-ai-suggester.ts`, `scope-analyst/run.ts`. Cada um tem seu próprio padrão (`messages.parse` vs `messages.create + tool_use`) — falta um wrapper comum.
- **Pricing refs loading** (productivity, labor rates, materials, divisions, trades) é replicado entre `accept-estimate/route.ts` (linhas 100-200) e `sync-from-takeoff/route.ts` — copy-paste sem extração.
- **Pattern `async function load() { ... } useEffect(() => load(), []); }`** repetido em ~10 components. Dá pra extrair pra um `useFetch` ou `useResource` hook customizado.

`grep -rn "TODO|FIXME|XXX|HACK" src/` → **zero matches**. Código limpo de marcadores. Pode ser positivo (ninguém deixou debt comentado) ou negativo (debt existe mas não foi marcado — vide tudo acima).

### 7.4 Risco de breaking changes (`npm install` + `npm run build`)

#### `npm install` — **probabilidade Alta de funcionar**

- `package.json` consistente, `package-lock.json` deve estar versionado (não verificado mas presumido), versões pinadas razoáveis.
- ⚠️ **`next-auth: ^5.0.0-beta.31`** — beta. `^5.0.0-beta.31` aceita qualquer beta minor/patch posterior do major 5; bumps de beta podem quebrar API.
- ⚠️ **`shadcn: ^4.3.0`** — major 4. Sem garantia de runtime estável.
- ⚠️ **`radix-ui: ^1.4.3`** — pacote umbrella relativamente novo. Comporta-se OK historicamente.
- ✅ Anthropic SDK 0.90, Prisma 6.19, Tailwind 4 — tudo majors estáveis.

#### `npm run build` — **probabilidade Média de funcionar**

Razões pra dúvida:

1. **7 TS errors atuais** (Buffer typing). Next normalmente ignora type errors em produção, mas se `next.config.ts` tiver `typescript.ignoreBuildErrors: false` (default) e algum upgrade de Next mudar comportamento → build falha. Atual `next.config.ts` é vazio, então usa default.

2. **`middleware.ts` deprecated** — o dev log mostra: `⚠ The "middleware" file convention is deprecated. Please use "proxy" instead.` Em algum minor bump de Next 16 isso pode virar erro.

3. **App Router em version `16.2.4`** — bem recente. O briefing alerta repetido: "_This is NOT the Next.js you know — read `node_modules/next/dist/docs/` first_". Build pode pedir migração de algumas APIs deprecated.

4. **Anthropic Files API beta** — `client.beta.files.upload` e `client.beta.messages.create` — estáveis hoje, mas `betas: ['files-api-2025-04-14']` header pode mudar slug a qualquer momento.

5. **Prisma generate precisa rodar antes** — em CI/Vercel o build script teria que rodar `prisma generate` no postinstall. Atual `package.json` **não tem postinstall hook**. Em local funciona porque Andre rodou manualmente. Em Vercel deploy isso quebra: `Cannot find module '.prisma/client'`. **Provável bloqueio em Vercel.**

6. **Buffer types do Node 24** — os 7 errors atuais são desse mismatch. Se Next/Vercel build fica em strict TS, falha.

#### Veredicto rápido

| Scenario                | Probabilidade                                                                                                                                                                    |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `npm install` local     | 🟢 **Alta** — 90%+                                                                                                                                                               |
| `npm run build` local   | 🟡 **Média** — provavelmente passa porque Next ignora TS errors em build, mas warnings de middleware deprecated + qualquer regressão dos 7 type errors podem morder              |
| Vercel deploy first try | 🔴 **Baixa** — sem postinstall hook pra `prisma generate`, vai falhar com "@prisma/client did not initialize yet". Fix: adicionar `"postinstall": "prisma generate"` no scripts. |
| `npm run dev` local     | 🟢 **Alta** — Andre roda diariamente sem issues maiores além de Neon cold start (não-bloqueador).                                                                                |

**Recomendações pra destravar build em Vercel:**

```json
// package.json
{
  "scripts": {
    "postinstall": "prisma generate",
    "build": "prisma generate && next build"
  }
}
```

E corrigir os 7 Buffer errors com cast explícito (`Buffer.from(arrayBuffer)`) — 30 min de trabalho.
